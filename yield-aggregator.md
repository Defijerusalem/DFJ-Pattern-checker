# Yield Aggregator Pattern Library

Validated against: Yearn legacy vault exploit ($9M, December 2, 2025) and Yearn yETH pool exploit ($9M, November 30, 2025) — two distinct incidents, three days apart, same protocol, different root causes. Also validated: GoGoPool ggAVAX (confirmed high-severity audit finding, mitigated) and Thetanuts Finance (~$50K, early 2026, newly-deployed vault) — two independent confirmations of the specific first-depositor inflation attack variant of Pattern 2, now properly evidenced rather than a bare edge-case mention. Dry-run tested against Beefy Finance (confirmed strong, tested-not-theoretical panic mechanism for Pattern 1; surfaced the underlying-protocol risk-inheritance rule now in SKILL.md) and Volo/Navi (Sui, Move-based — confirmed the failure category (access/key management, not exotic Move-specific logic) transfers cleanly across chain architectures; also a real example of a protocol pledging to absorb a loss rather than socializing it).

## Global evidence-quality standard

A clean NOT PRESENT requires a specific, checkable fact — a disclosed, executable kill-switch mechanism for deprecated strategies, or verified vault accounting logic that doesn't depend on raw, donatable balances. Vendor language ("actively managed," "rigorously monitored strategies," "battle-tested vaults") is never sufficient on its own. If only this kind of language is available, output SIMILAR MATCH, not NOT PRESENT.

## Before applying any pattern below: check for inherited risk from underlying protocols

A yield aggregator's vault wraps an underlying protocol (a lending market, a DEX pool, a staking contract). The aggregator's own contract can be clean while the underlying protocol it deposits into carries real risk. Before checking Patterns 1 and 2 below, identify what underlying protocol(s) the vault/strategy interacts with, and check that underlying protocol against the relevant other reference file (`lending.md` for a vault depositing into a lending market, `dex-amm.md` for a vault providing liquidity to a DEX pool, etc.). Any unresolved EXACT MATCH or SIMILAR MATCH finding on the underlying protocol should be carried forward and disclosed as inherited risk on the aggregator vault, even if the vault's own wrapping logic passes Patterns 1 and 2 cleanly. This mirrors how some platforms in this space explicitly flag underlying-platform risks as automatically inherited by every product built on top, regardless of whether the wrapping product separately discloses them.

## Pattern 1: Deprecated/Legacy Strategy Retirement Risk

**What to check:** whether the protocol has older, superseded vault versions or strategies that remain live and holding user funds, and whether governance has a genuinely *executable* path to force-retire them — not just a theoretical one.

**Why this matters — real case:**
- Yearn's December 2, 2025 exploit ($9M) targeted a known governance problem: who has authority and responsibility to decommission deprecated, vulnerable code. Early vault contracts had emergency shutdown mechanisms, but executing them required governance votes that never reached consensus — the vulnerable vaults simply continued existing, holding millions in user deposits, because the kill-switch existed on paper but couldn't actually be triggered in practice. The specific vulnerability involved how the deprecated vaults obtained price information (calling a DEX directly for prices, an outdated pattern superseded in newer versions).
- This is a process/governance failure, not a hidden code flaw — the vulnerability wasn't secret, the gap was that no one could act on the knowledge fast enough.
- Second confirming case, different category: Abracadabra Money's October 4, 2025 exploit (~$1.7M) hit an already-superseded CauldronV4 market in `lending.md` that had gone unaudited for 22 months while development moved to newer features — the same "deprecated but still live" mechanism, this time triggered by a code-logic bug rather than a stalled governance vote. See `lending.md`'s "What this file does NOT cover" section for the full case. Treat this pattern as validated across both Lending and Yield Aggregator.
- Third confirming case, outside either category: Truebit, a decentralized off-chain computation protocol (January 8, 2026, ~$26.4M) — an unprotected integer addition in a five-year-old, closed-source, never-publicly-audited token bonding-curve contract (compiled under Solidity 0.6.10, predating native overflow protection) allowed an attacker to craft an input that wrapped the computed mint price to zero, minting hundreds of millions of tokens for negligible cost. Truebit does not fit Lending, Yield Aggregator, or any of this library's eight categories cleanly, but the mechanism is the identical "deprecated, unaudited, still-live code as attack surface" precondition already validated twice above — confirming this is a genuinely category-agnostic risk, not specific to lending or yield products. A copycat second attacker minted a smaller amount (~$251K) within minutes of the primary exploit, reportedly from a wallet pre-funded via Tornado Cash days earlier; this is real, notable forensic context but describes attacker behavior and timing, not a checkable protocol design decision, and does not change what should be verified before deployment (operational status and audit history of any live legacy contract, per SKILL.md Step 1).
- A distinct, more severe variant worth checking separately, still under active investigation with several analyses circulating that partially conflict — this entry has been corrected twice already as more careful technical review has emerged, and is written with that history in mind rather than asserting settled confidence again: Summer.fi's Lazy Summer / FleetCommander (July 2026, ~$6.1M). FleetCommander is a meta-vault ("parent vault") that allocates deposited funds across multiple third-party protocols through strategy modules called "Arks" — one per integrated protocol (Morpho, Silo, and others), sitting alongside a VaultV2 component and a separate "Raft" contract managing rewards. **What is independently confirmed from the primary transaction trace and methodologically careful review:** the attacker flash-loaned USDC and USDT from Morpho, then — entirely within one transaction — called a sequence of public, legitimately-existing functions across FleetCommander, the Arks, and VaultV2 (deposit, redeem, withdrawFromArks, withdrawFromBuffer, forceDeallocate), exploiting how accounting and liquidity assumptions shifted between those calls within the same transaction. Of the contracts involved, twelve of thirteen have public, verified source code — including FleetCommander, the Arks, VaultV2, BufferArk, and the Morpho adapter itself — meaning the exploited logic sat inside the protocol's own public, previously-audited code, not inside a hidden or unverified component. Only the attacker's own orchestrating contract is unverified, and careful trace/bytecode analysis characterizes its role as sequencing and calling the public functions above in a specific order, rather than containing its own hidden protocol-level exploit logic. The attacker redeemed roughly $6.1M more than they deposited and repaid the flash loan within the same transaction. **What remains genuinely unconfirmed, though a fourth, more specific account has since narrowed the field:** the precise accounting flaw that made a sequence of public, legitimate calls exploitable. Two earlier analyses proposed a direct, attacker-timed donation event at the Ark level. A third, more methodologically careful analysis described a broader same-transaction accounting/liquidity-assumption exploit across all four components without confirming a specific single-donation point. A fourth account adds a real, checkable, and distinct causal layer: rather than the mispricing being purely attacker-manufactured within the exploit transaction, it proposes the inflated value originated from a genuine, pre-existing condition in an underlying Silo lending market (referred to as bUSDC-155) that had experienced prolonged utilization above 100% — meaning borrowed amounts exceeded what the pool could ever actually collect, generating real but uncollectable accrued interest that inflated that market's own share price from roughly 0.001 to 0.55 USDC over time, independent of the attacker's transaction. Under this account, the attacker acquired a governance/vault token (vgUSDC) that was already overvalued as a result of holding the mispriced bUSDC-155 position, then transferred it into SiloManagedVaultArk, whose balance-based accounting inherited the inflated value, which FleetCommander's LVUSDC vault in turn inherited from the Ark — allowing redemption at the inflated price. This account is notable for identifying a specific, well-understood, and independently checkable precondition (sustained over-100%-utilization bad debt inflating a lending market's share price) rather than attributing the entire mispricing to the exploit transaction alone, and it is compatible with, rather than contradictory to, the broader "nested vault trusts downstream share price without independent validation" finding already established above — it explains a plausible real source of the bad price this file's core finding describes. None of the four accounts have yet been superseded by a named security firm's formal, authoritative post-mortem. **How to check regardless of which specific account proves correct:** for any meta-vault or yield router built from multiple composable public functions across a parent and its strategy modules, verify whether calling those functions in an unusual sequence within a single transaction (deposits, redemptions, forced reallocations, buffer withdrawals) can create a window where the parent's accounting reflects a value that does not correspond to real, settled backing — same-transaction composability across otherwise individually-correct public functions is a real attack surface distinct from any single function having a bug in isolation.

**How to check:**
- Identify whether the protocol has multiple live vault/strategy versions, and whether older versions use outdated patterns (e.g., direct DEX price reads instead of TWAP, deprecated dependency versions).
- Check whether an emergency shutdown/deprecation mechanism exists, AND whether it has actually been used or tested — a mechanism that exists in code but has never been triggered, especially if it requires a governance vote with no enforced timeline, should not be treated as equivalent to a real, executable safeguard.
- Check published vault/strategy risk scores if the protocol maintains one (some established platforms publish internal risk-scoring for this exact purpose — treat the existence of such a score as a positive signal, but verify it's actually being acted on, not just published).

**Match criteria:**
- EXACT MATCH: deprecated/legacy strategies are confirmed live with user funds, use outdated/known-risky patterns, and the deprecation mechanism requires an unresolved governance process with no enforced deadline.
- SIMILAR MATCH: legacy strategies exist but a shutdown mechanism has either been used successfully before or has a disclosed, time-bound execution path.
- NOT PRESENT: no live deprecated strategies holding meaningful user funds, or all legacy strategies have been actively, successfully migrated/shut down.

## Pattern 2: Donation-Manipulable Vault Share-Price Math

**What to check:** whether a vault's share price (the exchange rate between deposited tokens and vault shares) is calculated from a raw, directly-donatable balance rather than an internally-tracked accounting value.

**Why this matters — related case (Lending category, same root mechanism):**
- This pattern is the same underlying mechanism validated in `lending.md` Pattern 2 via Cream Finance's October 2021 exploit, where a Yearn vault token's exchange rate could be manipulated by donating assets directly to the underlying vault. It applies equally here because yield aggregators are the primary issuers of this type of share-price-bearing vault token.
- Separately, Yearn's own November 30, 2025 yETH exploit ($9M) involved a related but distinct mechanism: manipulation of an iterative fixed-point solver used for LP token minting, causing invariant breakdown and arithmetic underflow. This is closer to a Universal Core code-logic issue (solver/math implementation flaw) than a donation-manipulation pattern specifically — flag separately and with lower confidence than Pattern 1 above.

**Specific, well-documented variant — first-depositor inflation attack:** the most common real-world instance of this pattern occurs at a vault's genesis, before any legitimate deposits exist. An attacker deposits a minimal amount (as little as 1 wei) to become the first depositor and sole share holder, then donates a large amount of the underlying asset directly to the vault (bypassing the deposit function entirely). This inflates the vault's `totalAssets()` relative to its `totalSupply()` of shares to an extreme ratio. The next legitimate depositor's share calculation then rounds down to zero shares due to integer division, meaning their deposit is fully absorbed with nothing returned — the attacker then withdraws, capturing the victim's deposit. Two independent, real, confirmed cases: GoGoPool's ggAVAX token (a high-severity finding in the protocol's own audited liquid-staking token, confirmed by the GoGoPool team and mitigated by seeding the vault with an initial deposit at initialization) and Thetanuts Finance (an earlier 2026 incident, ~$50,000, on a newly-deployed vault — a separate, later $2.1M incident hit a different, deprecated Thetanuts vault via an unrelated mechanism; see `lending.md`'s note on that second incident, correctly excluded as a Universal Core code-logic case). The standard, confirmed mitigation is OpenZeppelin's virtual shares/decimal offset approach, which makes the vault behave as if it always had a small amount of pre-existing assets and shares, preventing the ratio from ever reaching an exploitable extreme.

**How to check:**
- Locate the function that calculates share price or exchange rate.
- Check whether it derives from `balanceOf(vaultAddress)` (a raw balance anyone can inflate by sending tokens directly) versus an internally-tracked deposit ledger.
- For newly-deployed or newly-observed vaults specifically: check whether the vault seeds an initial deposit at deployment (dead shares, virtual offset, or an equivalent mechanism), or whether the first real depositor is exposed to the un-mitigated ratio described above.

**Cross-reference note:** as in `lending.md`, the practical attack here is typically flash-loan-funded — a large, momentary deposit is enough to distort the share price within one transaction before withdrawing. This pattern already fully covers that mechanism; do not score flash-loan funding as a separate, additional finding.

**Match criteria:**
- EXACT MATCH: share price is calculated from a raw, externally-donatable balance with no internal accounting check, OR a newly-deployed vault has no seeded initial deposit / virtual offset protection against the first-depositor inflation attack described above.
- SIMILAR MATCH: internal accounting exists but edge cases (e.g., solver-based calculations) are present and not independently verified as safe.
- NOT PRESENT: share price is calculated purely from an internally-tracked deposit ledger, immune to direct balance donation, AND (for newly-deployed vaults) a confirmed seeding/virtual-offset mechanism protects the first depositor.

## What this file does NOT cover

- Solver/iterative-math implementation bugs (the specific mechanism in Yearn's yETH incident) — this is closer to a novel code-logic flaw than a recognizable pattern; treat any finding here as a flag for professional audit review, not a confident pattern match, unless a second independent case validates a specific, checkable trigger.
- Strategy yield/APY sustainability and tokenomics — an economic design question, not a security pattern.
- Redemption/mint math integer-division truncation in deprecated or legacy vaults (Thetanuts Finance, June 15 2026, ~$2.1M — a separate, later incident from the earlier first-depositor case above, on a different, already-deprecated legacy vault: a flash-loan-funded burn-then-mint cycle drove the vault's supply near zero, causing required basket-token transfers per newly-minted share to round down to zero due to integer division, letting the attacker mint new shares requiring no real backing). This is the same Universal Core / code-logic boundary as Euler's `donateToReserves` bug and Osmosis's `JoinPoolNoSwap` bug — an arithmetic/rounding flaw in mint-time validation, not a structural architectural pattern this file is built to catch. Notably, a whitehat recovered the large majority of funds (~$2M of $2.1M) before the attacker could fully cash out, unrelated to the underlying vulnerability class.
