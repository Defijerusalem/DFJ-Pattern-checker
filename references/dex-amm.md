# DEX/AMM Pattern Library

Validated against: KiloEx ($7M, 2025, across Base/BNB Chain/Taiko), PancakeSwap BCE/USDT pool ($679K, March 2025), SwapNet and Aperture Finance (~$17M combined, January 25 2026, same day, same root cause — unrestricted low-level call-target validation combined with unlimited token approvals — two independent confirmations validating Pattern 3). Dry-run tested against Curve Finance (confirmed strong, manipulation-resistant EMA oracle design; surfaced the Harvest Finance downstream-integrator-misread distinction now built into Pattern 1, later reinforced by bZx as a second, earlier confirming case) and Osmosis (Cosmos-native; JoinPoolNoSwap accounting bug, $5M, June 2022 — confirmed correctly out-of-scope as a code-logic bug, not a category-specific pattern, same boundary as Euler). Cetus Protocol ($223M, May 2025, Sui — Move-based) was previously mis-cited as a Pattern 1 validating case; corrected — see "What this file does NOT cover" below.

## Global evidence-quality standard

A clean NOT PRESENT requires a specific, checkable architectural fact — confirmed TWAP usage in the actual pricing function, a named oracle integration, documented pool depth relative to typical trade size. Vendor language ("battle-tested AMM," "secure pricing mechanism," "audited and trusted") is never sufficient on its own. If only this kind of language is available, output SIMILAR MATCH, not NOT PRESENT.

## Pattern 1: Spot-Price-Only Pricing (No TWAP)

**What to check:** whether the protocol's pricing function reads the current-block spot price/reserve ratio directly, or uses a time-weighted average price (TWAP) that requires manipulation to be sustained across multiple blocks.

**Why this matters — real cases:**
- The general mechanism: an attacker uses a flash loan to fund a massive single-block swap that distorts an AMM pool's spot price, then exploits that distorted price elsewhere (in the same pool, or in another protocol that reads this pool's price) before reversing the trade in the same transaction. The cost of the attack is bounded by flash loan fees; the profit is bounded only by how much value depended on the manipulated price.
- KiloEx lost $7M across three chains in 2025, and PancakeSwap lost $679K, both to this category of oracle/price manipulation attack — trusting a manipulable instantaneous price.
- New Gold Protocol (September 17-18, 2025, ~$1.9-2M, BNB Chain) — a third, real confirming case, with a genuinely additional compounding mechanism worth checking separately. The token's `getPrice()` function calculated value by directly referencing current Uniswap V2 pair reserves — the standard Pattern 1 mechanism. But the protocol also had per-user buy limits and cooldown periods explicitly designed to prevent price-inflation attacks like this one. The attacker bypassed both safety checks entirely by routing the flash-loan-funded swaps through the zero/dead address, which had been whitelisted (alongside the token contract and a mint address) to receive amounts above the normal limits with no cooldown. This is worth checking as its own sub-question: even where a protocol has designed real transaction-level safety limits specifically to blunt this exact attack, check whether any whitelisted address (dead address, contract addresses, designated system wallets) is exempt from those same limits — an exemption meant for legitimate internal operations can become the exact bypass route for the attack the limits were built to prevent.
- Palmswap (July 24, 2023, ~$901K, BNB Chain — 80% later recovered through direct negotiation with the attacker) — a distinct, more fundamental variant of the same theme: the safeguard was never bypassed, because it was never active in the first place. The `PlpManager` contract calculated an asymmetric PLP-to-USDP exchange rate — a flash-loan-funded add-then-remove-liquidity cycle exploited a rate mismatch (effectively 1:1 on entry versus roughly 1.9:1 on exit for the same underlying value), letting the attacker extract far more than they deposited in a single transaction. The protocol had explicitly built a `cooldownDuration` mechanism intended to prevent exactly this kind of same-block or rapid liquidity-cycling attack — but that duration parameter had been deployed and left at its default value of zero, meaning the cooldown imposed no actual delay at all. This is a distinct lesson from New Gold Protocol's bypass-via-whitelist-exemption: here, check whether a safety parameter's *configured value*, not just its presence or exemption list, is set to something that actually functions as intended — a cooldown, threshold, or limit deployed at its inert default (zero, unlimited, or otherwise a no-op value) provides no protection even though the mechanism technically "exists" in the code.
- Gamma Strategies (January 4, 2024, $3.4-6.4M across multiple Arbitrum vaults on Uniswap, Camelot, and Ramses) — a distinct, valuable variant: a real, deliberately designed safeguard existed, but was miscalibrated rather than absent. The protocol's Hypervisor vaults calculated LP shares from real-time pool prices, and explicitly implemented four layers of flash-loan protection, including a maximum allowed price-change threshold before a deposit would be rejected. That threshold, however, had been configured to permit a 50% to 200% price swing on certain LST and stablecoin vaults — far looser than the assets' normal volatility warranted. The attacker used flash loans to push the price to the edge of that overly generous threshold, minted a disproportionate number of LP shares at the inflated price, then reversed the price and withdrew far more value than deposited, repeating the cycle. This is the same core mechanism as Pattern 1 above, but the lesson is distinct: the presence of a price-change safeguard is not sufficient — the specific numeric threshold chosen must be checked against the actual expected volatility of each asset pair, since a technically-present but loosely-set limit provides a false sense of protection while still leaving the door open.
- Cetus Protocol's $223M loss (May 2025) is commonly cited alongside these cases but does not actually validate this pattern — see the correction in "What this file does NOT cover" below before citing Cetus as a Pattern 1 example.

**How to check:**
- Locate the pricing function in the contract (often named something like `getPrice`, `_getReserves`, or similar).
- Check whether it reads `reserve0/reserve1` (or equivalent) directly from the current block, or whether it averages price over a window of blocks/time (TWAP).
- Check pool depth (total liquidity) relative to typical trade sizes the protocol expects — a thin pool is manipulable with less capital.

**Cross-reference note:** this pattern already fully covers flash-loan-funded price manipulation as its core, validated mechanism (see KiloEx above) — do not score flash-loan risk as a separate, additional finding for DEX/AMM protocols. A single flash-loan-funded attack should be scored once, against this pattern.

**Match criteria:**
- EXACT MATCH: pricing function uses current-block spot price with no time-averaging, AND this price is used as an oracle input for another function (lending, liquidation, derivative settlement) either within the same protocol or by an external integrator.
- SIMILAR MATCH: TWAP exists but the averaging window is short enough to be questionable (e.g., under 10 minutes), or TWAP implementation details are undisclosed.
- NOT PRESENT: confirmed TWAP or equivalent manipulation-resistant pricing with a documented, reasonable time window (verify in code, not just claimed in docs).

**Critical distinction — check this separately, even when Pattern 1 above clears NOT PRESENT:** a protocol's own pricing mechanism being manipulation-resistant does not mean every consumer of that pool's data is safe. Harvest Finance was exploited for ~$500K (in repeated $50M flash-loan cycles) by reading a Curve pool's *live reserve ratio* directly, instead of Curve's actual manipulation-resistant oracle value — Curve's own pricing was never at fault, but a downstream integrator read the wrong number. An earlier, founding case of the same underlying trust failure predates Harvest by eight months: bZx (February 2020, two separate incidents, ~$650K and ~$633K) was a lending protocol that trusted Kyber's spot price for collateral valuation — Kyber itself routed large orders through Uniswap's thin reserves, so a flash-loan-funded trade on Uniswap directly inflated the price bZx read for collateral, in both a wBTC-collateral attack and a separate sUSD-collateral attack four days later. This is a lending protocol consuming an external, manipulable spot price for collateral valuation — mechanistically the same trust failure as Harvest's downstream reserve-ratio read, just from the lending side rather than the yield-vault side, and confirming the pattern was never a one-off. When checking any protocol that consumes price data from an external pool (rather than using its own internal pricing), explicitly verify it reads the smoothed/resistant oracle output, not a raw spot balance or reserve ratio. Standard audits often miss this because it's an architectural/integration choice, not a bug in either contract individually. Score this as a SIMILAR or EXACT MATCH against Pattern 1 even if the source pool itself is well-designed, if the integration method reads the unprotected value.

Also check for asset-specific oracle gaps: some manipulation-resistant oracles do not automatically account for yield-bearing or rebasing wrapped tokens (e.g., an oracle may return the underlying asset's price rather than a yield-bearing derivative's actual value including accrued yield). If the protocol integrates such an asset, verify the integration manually accounts for the conversion rate — an unverified assumption here is a SIMILAR MATCH, not NOT PRESENT.

## Pattern 2: Pool Ratio Distortion via Auxiliary Mechanism (Burn/Fee/Donation)

**What to check:** whether any auxiliary pool mechanism (token burn, fee redistribution, reward distribution) can be triggered or amplified by an external actor in a way that distorts the pool's price ratio independent of an actual trade.

**Why this matters — real case:**
- PancakeSwap's March 2025 BCE/USDT exploit: attackers deployed malicious contracts that bypassed buy/sell limits and manipulated the pool's token burn mechanism, artificially distorting the BCE/USDT ratio to create an arbitrage opportunity — a $679K loss. This is distinct from Pattern 1: the manipulation vector was the burn mechanism itself, not a flash-loan-funded spot price swing.

**How to check:**
- Identify any non-swap mechanism that alters pool token balances (burns, fee skims, auto-compounding, rebasing).
- Check whether that mechanism can be triggered by an external caller, or whether its trigger conditions can be gamed (e.g., triggering a burn repeatedly in quick succession before the pool can rebalance via arbitrage).

**Match criteria:**
- EXACT MATCH: an externally-triggerable auxiliary mechanism exists that directly alters pool ratio and has no rate-limiting or access control.
- SIMILAR MATCH: mechanism exists with some access control or rate-limiting, but the limit's adequacy is unverified.
- NOT PRESENT: no externally-triggerable balance-altering mechanism beyond standard swap function, or such mechanisms are restricted to a verified, access-controlled role.

## Pattern 3: Unrestricted Call-Target Validation in Swap Aggregators (Arbitrary Call / Unlimited Approval Combination)

**What to check:** for DEX aggregators and liquidity-position managers that execute a low-level call to a user-supplied or dynamically-determined address as part of a swap or position-management routine, whether the target address and function selector are validated against a whitelist before the call executes — and separately, whether the protocol encourages or defaults to unlimited/infinite token approvals rather than per-transaction limited approvals.

**Why this matters — two independent, real, dollar-quantified cases, same root cause, same day:**
- SwapNet (January 25, 2026, ~$13.4M): the router's core swap-routing function performed a low-level call to a user-controlled address with user-controlled calldata, for one of its supported "custom router" swap types, with no validation of the call target. An attacker crafted calldata that pointed the call at the USDC contract itself and encoded a `transferFrom` call, draining every user who had granted the router an infinite token approval. Closed-source, unverified bytecode compounded the risk by preventing community review.
- Aperture Finance (same day, January 25, 2026, ~$3.67M): an unrelated but mechanistically identical vulnerability in its Uniswap V3 position-management contract — an internal function executed a low-level call using user-supplied calldata with no restriction on the call target or function selector, allowing the same class of attack: siphoning ERC-20 tokens and even approving away Uniswap V3 position NFTs.
- Kame Aggregator (September 13, 2025, ~$1.3M, Sei Network) — the earliest confirmed case of this exact mechanism, predating SwapNet and Aperture Finance by four months. The router's swap-execution function let the caller set an arbitrary "executor" contract; the attacker pointed it at a public Multicall contract and used it to directly invoke `transferFrom()` against any of the roughly 830 users who had granted the router a token approval, regardless of whether they had an active trade in progress. Confirms the same combination described above was already a known, real risk more than a year before this reconciliation pass — no whitelist on the executor address, and unlimited/oversized approvals left standing after use, turning a convenience feature into a standing liability.
- Socket Protocol/Bungee Exchange (January 16, 2024, ~$3.3M) — the true founding case of this mechanism, roughly 20 months before Kame. A newly-added bridging route's `performAction()` function passed a user-supplied `swapExtraData` parameter directly into a low-level call with no validation. The attacker injected a `transferFrom` call into that parameter, draining any of the roughly 230 wallets that had granted the Socket Gateway contract an infinite token approval — the exact same "arbitrary call target/calldata combined with unlimited approval" combination described above, just expressed through unvalidated calldata content rather than an unvalidated destination address. Confirms this mechanism has been real and repeating for over two years, not a recent phenomenon specific to any one wave of aggregators. Fixed within 14 minutes of discovery, but the two transactions executed before the fix were enough to complete the drain — reinforcing that detection speed alone cannot substitute for the validation gap being absent in the first place.
- Cross-category reinforcement, outside this file's scope: HyperDrive (September 27, 2025, ~$782K), a lending protocol on Hyperliquid rather than a DEX aggregator, lost two liquidity pools to the identical underlying mechanism — users had granted the protocol's Router excessive, standing operator permissions over its whitelisted contracts, and an attacker exploited an arbitrary call within that trusted router path to repeatedly withdraw funds. A second, independent confirmation of the same general mechanism: a Yearn-adjacent third-party integrator contract (April 28, 2026, ~$1M in yvWETH) held a victim's unlimited (`uint256.max`) token approval, and exposed a completely unauthenticated public `execute(Action[])` batch-dispatcher function with no caller restriction — any address could call it directly, weaponizing the standing approval to pull and redeem the victim's vault shares. Yearn's own vault contract (yvWETH) was not flawed; the vulnerability was entirely in a separate, third-party contract that happened to hold an approval and exposed arbitrary execution to any caller. Neither of these is filed as a validating case for Pattern 3 above, since neither is a swap aggregator this file covers specifically — but together they confirm "excessive standing approval combined with an arbitrary or unauthenticated execution path" is a real, recurring, cross-category mechanism, not unique to swap routing or any single protocol type. **How to check, regardless of category:** for any contract holding a standing token approval on a user's behalf, verify that every function capable of moving or redeeming that approved balance has genuine caller access control — not merely being embedded inside a larger, seemingly-trusted router or integrator contract.
- A distinct, real lesson worth checking separately: verifying the caller's identity is necessary but not sufficient — the payload itself must also be authenticated. 1inch TrustedVolumes (May 7, 2026, ~$5.87M, and confirmed by security researchers as the same actor behind the March 2025 Fusion v1 incident above) exploited market-maker resolver bots that checked only whether an incoming call's `msg.sender` matched the legitimate 1inch settlement contract's address — a correct but incomplete check. The attacker found a way to route a forged, malicious call *through* the genuine settlement contract itself, so the sender-address check passed even though the underlying instruction was fraudulent. The bots trusted the messenger without verifying the message. **How to check:** for any contract that restricts a sensitive function to calls originating from a specific, trusted address, confirm the check also validates that the *content* of the call is legitimate and expected — not merely that it arrived from the correct sender, since a trusted contract can itself be manipulated into forwarding an attacker's forged instruction.
- A related but mechanistically distinct library-level flaw: combining the ERC2771 meta-transaction standard with a multicall/batch-call function can allow an attacker to impersonate any address as the transaction's "true sender." Floor Protocol (December 2023, ~$1.6-2M in stolen NFTs) suffered exactly this: its trusted-forwarder logic appended the real sender's address to the end of calldata for context, per the ERC2771 standard, but a flawed contract upgrade allowed this mechanism to be combined with an unrestricted multicall function. An attacker could craft calldata whose final 20 bytes matched a victim's address, tricking the contract into executing `transferFrom` as if the victim themselves had authorized it — no real approval or signature from the victim was required beyond a pre-existing token approval. This was not unique to Floor Protocol: the same combination of ERC2771 plus multicall was independently confirmed to affect Thirdweb's own shared library implementation, which in turn put Coinbase NFT, OpenSea, and other major platforms building on the same pattern at risk before it was patched. **How to check:** if a contract implements ERC2771 (or any meta-transaction/trusted-forwarder standard) alongside a multicall or batch-execution function, verify that the trusted-forwarder context length is validated and cannot be manipulated by attacker-supplied calldata, and confirm the specific library version in use has been checked against this known vulnerability class.

**The specific combination that makes this dangerous:** neither the arbitrary-call flaw nor the unlimited-approval pattern is dangerous alone. An arbitrary call with no prior approvals granted has nothing to drain. An unlimited approval to a contract with a properly whitelisted call path is safe. The combination — an aggregator that both (a) accepts broad, unvalidated call targets in its execution path and (b) is commonly granted infinite/unlimited approvals by users for convenience — is what turns a code-level flaw into a mass-drain event across every user who ever approved the contract, not just users active at attack time.

**How to check:**
- Identify whether the protocol's swap-execution or position-management logic performs any low-level call (`call`, `delegatecall`) to an address that is either user-supplied or determined by data the user controls.
- If so, check whether that target address is validated against an explicit whitelist of known, legitimate router/pool contracts, and whether the function selector being invoked is restricted to an expected set (e.g., only swap-related selectors, never `transferFrom` or `approve`).
- Check whether the protocol's documentation or default integration pattern encourages unlimited/infinite approvals rather than per-transaction, limited approvals.
- Check whether the contract source is verified and publicly readable — closed-source, unverified bytecode was a contributing factor in both validating cases, since it prevented community security review before exploitation.

**Match criteria:**
- EXACT MATCH: the protocol performs a low-level call to a user-controlled or dynamically-resolved address with no whitelist validation of the target, AND the protocol's normal usage pattern involves unlimited/infinite token approvals.
- SIMILAR MATCH: an unrestricted call-target pattern exists but the protocol's approval model is limited/per-transaction by default (reducing blast radius), or a partial whitelist exists that has not been independently verified as complete.
- NOT PRESENT: all low-level calls in the swap/position-management path are validated against an explicit, confirmed whitelist of legitimate targets and function selectors, or the protocol does not perform any user-directed low-level calls at all.

## Pattern 4: Fee-on-Transfer/Deflationary Token Hook Directly Corrupts Its Own AMM Pair Reserves

**What to check:** whether a token's own custom `_transfer`/`_update` override directly burns tokens out of, or mints tokens into, its own AMM pair's balance — or otherwise mutates the pair's token balance — outside of a genuine swap, and whether the resulting desync between the pair's tracked `reserve0`/`reserve1` and its actual token balance is then realizable by anyone calling `sync()`/`skim()` (both permissionless on a stock Uniswap-V2-style pair).

**Why this matters — real cases:** this is the single most repeated smart-contract exploit mechanism observed in a large-scale reconciliation pass across on-chain incident data (60-80+ independent instances follow this identical shape, overwhelmingly BSC/PancakeSwap "tax token" launches, individually modest in size but collectively the dominant failure mode by volume). A representative, dollar-quantified sample:
- SafeMoon V1 (~$8.9M, the entire WBNB side of its SafeSwap pair) — the implementation exposed `burn(address from, uint256 amount)` as a fully public function with no access control, letting anyone burn the pair's own token balance directly, collapsing the constant-product ratio before draining the other side.
- FPC Token (~$4.7M) — the sell-path `burnLpToken()` routine drained the pool's own FPC reserve directly rather than routing the sale through the pool.
- Caterpillar Coin/CUT (~$1.26M) — a bespoke "LP-removal value preservation" mechanic minted free tokens whenever liquidity was removed, rather than actually preserving value.
- Planet Finance (~$400K) — a zero-amount transfer branch triggered an uncompensated pool-side burn, meaning the exploit required no real capital to trigger, only a correctly-shaped zero-value call.
- P719 Token (~$312K) and NORMIE (~$490K) — both combine an uncompensated in-transfer burn with a `skim()` call that instantly realizes the artificially-collapsed reserve as extractable value for whoever calls it.

**The underlying reason this recurs so often:** a standard Uniswap-V2-style pair's `sync()` (recalibrate `reserve0`/`reserve1` to match actual balances) and `skim()` (send any balance in excess of tracked reserves to a caller-chosen address) are both intentionally permissionless by design — safe for a well-behaved token, but a direct weapon against any token whose own transfer logic can be tricked into moving balance into or out of the pair address outside the swap path.

**How to check:**
- Locate the token's custom `_transfer`/`_update`/`transferFrom` override (a plain, unmodified OpenZeppelin ERC-20 cannot exhibit this pattern).
- Check whether any branch of that override — a "sell tax," "deflation," "reflection," "auto-burn," "dividend," or similar mechanic — directly calls `burn`/`_burn`/a raw balance decrement against the pair's own address, or transfers/mints tokens directly to the pair, outside of the pair's own `swap()` function.
- Check whether `sync()` and/or `skim()` on the pair are freely callable by any address (true for a stock Uniswap-V2 pair; some forks add restrictions — confirm rather than assume).
- A closely related, smaller-scale variant worth checking in the same pass: whether the same custom `_transfer` reads the sender's and recipient's balances into local variables and writes them back independently, without special-casing `sender == recipient` — a self-transfer can then double-credit the recipient side without debiting the sender side (confirmed real, smaller-scale cases: GPU Token ~$32K, APIG ~$169K across two pools, LABUBU ~$12K, LPC Token ~$45.7K). Same root cause family (custom transfer logic that doesn't preserve the token's own conservation invariant), different trigger.

**Match criteria:**
- EXACT MATCH: the token's transfer hook directly manipulates the AMM pair's own token balance (burn-from-pair, mint-to-pair, or an uncompensated balance change touching the pair) outside the swap path, AND `sync()`/`skim()` on the pair is unrestricted.
- SIMILAR MATCH: the transfer hook manipulates the pair's balance but `sync()`/`skim()` access is restricted, delayed, or conditioned in a way that meaningfully narrows the exploit window.
- NOT PRESENT: the token uses an unmodified, standard transfer/swap path with no direct manipulation of any AMM pair's balance, confirmed by reading the actual `_transfer` override (or confirming none exists).

## What this file does NOT cover

- MEV/sandwich attack exposure at the user-transaction level — this is a real, distinct risk category but operates on individual trades rather than protocol-level pool integrity; treat as a separate, lower-severity note rather than a pattern match here unless a specific case validates point allocation.
- Impermanent loss as a risk to LPs — this is a known, disclosed economic property of AMMs, not a security vulnerability, and should not be penalized as if it were a flaw.
- Balancer V2 ($120-128M, November 3 2025, Composable Stable Pools across Ethereum, Arbitrum, Base, Polygon, Sonic) is a Universal Core / code-logic case, not a DEX/AMM structural pattern. Two independent, confirmed root causes: an arithmetic precision/rounding error in the `_upscaleArray` scaling function (unidirectional rounding during upscaling versus bidirectional rounding during downscaling, exploited via 65+ compounding micro-swaps within a single `batchSwap` to suppress BPT price toward zero), and separately a faulty access-control check in `manageUserBalance` where the contract compared `msg.sender` against a caller-supplied `op.sender` value, letting the attacker impersonate any account. Both are code-level logic/arithmetic flaws invisible to a pricing-mechanism or pool-ratio review — the pool's actual pricing design (Curve-style StableSwap invariant) is sound; the flaw was in how balances were scaled and validated internally. Despite 10+ audits from firms including OpenZeppelin and Trail of Bits, the bug went uncaught. Same boundary as Euler Finance and Cetus Protocol: this is not a category-specific structural pattern this file is designed to catch, and should not be cited as a DEX/AMM Pattern 1 or Pattern 2 validating case.
- KyberSwap Elastic (~$47-54M, November 23, 2023) — an even earlier, larger, and more thoroughly documented confirming case of the same boundary. Multiple independent security firms (CertiK, BlockSec, AuditOne, Halborn) traced the root cause to a precision/rounding discrepancy between two different internal calculations of the same tick-crossing logic in KyberSwap's concentrated liquidity math (`computeSwapStep` and `_updateLiquidityAndCrossTick`), causing the pool to believe it held more liquidity than it actually did. This is not a pricing-source trust decision or an auxiliary-mechanism design choice — it is a subtle, code-level arithmetic bug in how the AMM's own internal liquidity accounting handled a boundary condition, missed by prior audits. The attacker later demanded control of the company itself in exchange for returning funds, an unusual escalation with no bearing on the underlying technical classification. Same boundary as Balancer, Cetus, and Euler — do not cite as a Pattern 1 or 2 validating case.
- 1inch Fusion v1 (~$5M, March 2025) — a distinct variant of the deprecated-code risk already noted elsewhere in this library (Yearn, Abracadabra, Truebit): a low-level EVM memory-corruption bug (a buffer overflow corrupting the `interactionLength` variable used to locate a resolver's address within serialized calldata) in a contract that had been formally deprecated in 2023 but never disabled, allowing an attacker to forge a legitimate resolver's identity and swap negligible value for millions in real tokens. The specific new lesson: deprecated code is not just an accounting or governance liability — it can carry undiscovered low-level memory-safety bugs specifically because it stopped receiving security review attention the moment it was marked deprecated, even though it remained fully executable. Same Universal Core boundary as the cases above; do not cite as a Pattern 1, 2, or 3 validating case.
- **Correction — Cetus Protocol ($223M, May 2025, Sui) is a Universal Core / code-logic case, not a Pattern 1 case.** Cetus was previously cited as a Pattern 1 validating incident throughout this project; this was a mis-citation, caught during a later reconciliation pass, not confirmed during the original dry-run testing. The actual root cause (per SlowMist's published analysis) was an arithmetic overflow-check failure in a function called `checked_shlw`, which failed to reject a crafted input and let the attacker mint massive liquidity for a near-zero deposit. The attacker never manipulated a price feed or read a manipulable reserve — Pattern 1's actual mechanism. This is the same category of flaw as Osmosis's `JoinPoolNoSwap` bug and Euler's missing `donateToReserves` health check, both already correctly excluded above: a checked-math or accounting bug in the protocol's own logic, not a pricing-trust decision a pattern check is designed to catch. A pattern check run against pre-exploit Cetus using Pattern 1's actual criteria would plausibly have returned NOT PRESENT or SIMILAR MATCH at best, missing the real vulnerability — the same blind spot SKILL.md already documents for Euler ("six professional audits missed it"). Do not cite Cetus when explaining or validating Pattern 1.
